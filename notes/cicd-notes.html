01
What is CI/CD?
The problem it solves — and why every serious team uses it
Key Concept

CI (Continuous Integration) means every time a developer pushes code, an automated pipeline immediately runs the tests. If any test fails, the team knows within minutes — not days. CD (Continuous Deployment) means if all tests pass, the code is automatically deployed to production. No manual steps, no "who forgot to deploy?" — it just happens.

The problem without CI/CD

Without CI/CD — what actually happens:
 Dev A finishes a feature on Friday → pushes to main
 Breaks Dev B's feature that was working fine
 Nobody runs tests manually → bug reaches production
 Users complain → team debugs at 11pm on Friday night

With CI/CD:
 Dev A pushes → pipeline runs tests automatically → tests fail
 PR is blocked from merging → Dev A fixes it before it ever hits main ✅
 Tests pass → code auto-deploys → everyone goes home on time ✅
CI — Continuous Integration
Catch bugs early

Run lint, tests, and build checks automatically on every push or PR. Broken code never reaches the main branch.

CD — Continuous Deployment
Ship automatically

When tests pass on main, automatically deploy to staging or production. No manual SSH, no "can someone deploy this?"

03
GitHub Actions
How it's set up — one folder, one YAML file, automatic
Key Concept

GitHub Actions workflows live in .github/workflows/ inside your repo. When you push a .yml file there, GitHub detects it and runs it automatically on the events you specify. No external service to connect — it's built into your repo.

folder structure
your-project/
├── .github/
│   └── workflows/
│       ├── ci.yml        ← runs tests on every push/PR
│       └── deploy.yml    ← deploys when tests pass on main
├── src/
├── package.json
└── ...
You push code to GitHub
Any branch, any commit. GitHub detects the push event.
GitHub finds matching workflow files
Checks .github/workflows/ for any .yml with on: push. Queues them to run.
GitHub spins up a fresh Ubuntu VM (runner)
Clean slate every single run. No leftover state from previous runs.
Workflow steps execute in order
Checkout code → install Node → install deps → run tests → build.
Result posted to GitHub
Green checkmark ✅ or red X ✗ on the commit and PR. PR can be blocked from merging if the check fails.
04
Workflow YAML Syntax
Every key you'll use — explained line by line
annotated workflow skeleton
name: CI Pipeline           # Display name in GitHub UI

on:                          # What triggers this workflow
  push:
    branches: [main, develop]  # Only on pushes to these branches
  pull_request:
    branches: [main]           # Also on PRs targeting main

jobs:                        # All the jobs in this workflow
  test:                      # Job name (you choose this)
    runs-on: ubuntu-latest  # Which OS to use

    steps:                   # Steps run top to bottom
      - name: Checkout code
        uses: actions/checkout@v4   # Clones your repo onto runner

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci              # ci = clean install (faster, deterministic)

      - name: Run tests
        run: npm test

Key fields explained

FieldWhat it doesValues
on Triggers that start the workflow push, pull_request, schedule, workflow_dispatch (manual)
runs-on The OS of the runner VM ubuntu-latest (most common), windows-latest, macos-latest
uses A pre-built Action from GitHub Marketplace actions/checkout@v4, actions/setup-node@v4
run A raw shell command to execute npm ci, npm test, npm run build, any bash command
with Parameters passed to a uses action node-version: '20', cache: 'npm'
env Environment variables for a step or job NODE_ENV: test, DATABASE_URL: ${{ secrets.DB_URL }}
needs Makes a job wait for another job to finish first needs: test (deploy job waits for test job to pass)
if Conditional — only run this job/step if condition is true if: github.ref == 'refs/heads/main'
💡

Use npm ci instead of npm install in CI pipelines. It installs exactly what's in package-lock.json, never updates it, and is faster. npm install can silently upgrade packages and cause inconsistent builds.

05
Your First CI Pipeline
Test on every PR — blocks merge if tests fail

This is the core CI pipeline. Every time someone opens a PR targeting main, GitHub runs the tests. If tests fail, the PR gets a red X — it cannot be merged until fixed.

Create the workflow file
.github/workflows/ci.yml
name: CI — Test on PR

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    name: Run Jest Tests
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'         # cache node_modules between runs

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test
        env:
          NODE_ENV: test

      - name: Upload coverage report   # optional but useful
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/
Enforce it — Branch Protection Rule

The pipeline only blocks merges if you set up a branch protection rule in GitHub:

GitHub Settings → Branches → Add Rule
Go to: Settings → Branches → Add branch protection rule

Branch name pattern: main

Check these options:
✅ Require status checks to pass before merging
   → Search for and add: "Run Jest Tests"  (your job name)
✅ Require branches to be up to date before merging
✅ Do not allow bypassing the above settings
Dev opens PR: feature/add-login → main
    ↓
CI pipeline triggers automatically
    ↓
npm test → 2 tests fail
    ↓
PR shows ✗ "Run Jest Tests" — Merge button is greyed out
    ↓
Dev fixes tests, pushes again → pipeline re-runs
    ↓
All tests pass → ✅ green check → Merge button enabled ✅
06
Deploy Pipeline
Test first, then deploy automatically — only on main
Key Concept — needs

The needs keyword makes one job depend on another. The deploy job won't start until the test job finishes successfully. If tests fail, deploy never runs. This is the entire safety guarantee of CD in a single keyword.

.github/workflows/deploy.yml
name: CD — Test & Deploy

on:
  push:
    branches: [main]    # only runs when code lands on main

jobs:

  # ── Job 1: Run tests ─────────────────────────────
  test:
    name: Run Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npm test

  # ── Job 2: Deploy (only if tests passed) ─────────
  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: test              # ← waits for test job to succeed
    if: github.ref == 'refs/heads/main'

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Render / Railway / VPS
        run: |
          # Option A — trigger a Render deploy hook
          curl -X POST "${{ secrets.RENDER_DEPLOY_HOOK }}"

          # Option B — SSH into your VPS and pull + restart
          # ssh user@server "cd /app && git pull && pm2 restart all"

Common deploy targets

PlatformHow to deploy from ActionsComplexity
Render Trigger deploy webhook URL with curl ✓ Easiest — one curl command ✓
Railway Use Railway GitHub integration — auto-deploys on push ✓ Easiest — no workflow needed ✓
Vercel Use vercel/action from Marketplace Easy ✓
VPS (DigitalOcean/EC2) SSH in, git pull, pm2 restart Medium — need to set up SSH keys as secrets
Docker + ECR Build image, push to registry, update container Advanced — full production setup
07
Secrets & Environment Variables
Never hardcode credentials — store them in GitHub Secrets
⚠️

Never put API keys, DB passwords, or tokens directly in workflow YAML files. YAML files are committed to your repo. If your repo is public — or ever becomes public — those credentials are exposed. GitHub Secrets are encrypted and only injected at runtime.

Step 1 — Add secrets to GitHub
GitHub Settings → Secrets
Go to: Your Repo → Settings → Secrets and variables → Actions
Click "New repository secret"

Add these secrets:
  DATABASE_URL      = mongodb+srv://user:pass@cluster...
  JWT_SECRET        = your_super_secret_jwt_key
  RENDER_DEPLOY_HOOK = https://api.render.com/deploy/srv-xxx...
Step 2 — Use secrets in workflow
.github/workflows/ci.yml — using secrets
steps:
  - name: Run tests
    run: npm test
    env:
      # Secrets injected as environment variables at runtime
      DATABASE_URL: ${{ secrets.DATABASE_URL }}
      JWT_SECRET:   ${{ secrets.JWT_SECRET }}
      NODE_ENV:     test

  - name: Deploy
    run: curl -X POST "${{ secrets.RENDER_DEPLOY_HOOK }}"
💡

GitHub automatically masks secrets in logs. Even if a step accidentally prints process.env.DATABASE_URL, GitHub replaces the value with *** in the action logs. The actual value is never visible.

Environment-specific variables

Different values per environment
jobs:
  test:
    environment: staging    # uses staging secrets
    env:
      DATABASE_URL: ${{ secrets.STAGING_DB_URL }}

  deploy:
    environment: production # uses production secrets
    needs: test
    env:
      DATABASE_URL: ${{ secrets.PROD_DB_URL }}
08
Best Practices
What separates a real CI/CD pipeline from a toy one

Cache dependencies to speed up builds

By default, the runner installs all node_modules from scratch every run — slow. Enable caching and subsequent runs skip the install if package-lock.json hasn't changed.

Caching node_modules
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'npm'    # ← one line — cuts install time from 60s to 5s

Run jobs in parallel to save time

Parallel jobs — lint and test simultaneously
jobs:
  lint:                           # runs at the same time as test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint

  test:                           # runs at the same time as lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test

  deploy:
    needs: [lint, test]            # waits for BOTH to pass
    runs-on: ubuntu-latest
    steps:
      - run: echo "deploying..."

Summary of best practices

PracticeWhy it matters
Use npm ci not npm install Deterministic installs — same packages every run, never silently upgrades
Cache node_modules Cuts build time from 60–90s to under 10s on repeat runs
Use secrets for credentials Never expose DB URLs or API keys in YAML committed to git
needs: test before deploy Deploy only runs if tests pass — the core safety guarantee
Pin action versions (@v4) Prevents a third-party action update from silently breaking your pipeline
Branch protection rules Actually enforces the CI check — without it, devs can merge failing PRs
Separate CI and CD workflows ci.yml runs on every PR. deploy.yml runs only on main. Cleaner and easier to debug.
if: always() on upload steps Upload test reports and artifacts even when tests fail — you need the report most when things break
The mental model

CI/CD is not a DevOps luxury — it's the minimum standard for professional software development. The pipeline is your safety net. Every company you interview at will have one. Building and maintaining pipelines is a skill that signals seniority. If you can set up CI/CD for your own projects, you'll stand out in every technical interview.
