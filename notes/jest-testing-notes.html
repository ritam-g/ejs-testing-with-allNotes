01
Types of Testing
Four categories — know what you're writing before you write it
Type 1
Unit Testing

Tests a single function or component in complete isolation. No database, no network, no dependencies. Fast and precise. Does this function return the right value?

Type 2
Integration Testing

Tests how multiple units work together. Example: does the route handler correctly call the service, which correctly calls the DB model? Do the pieces fit?

Type 3
End-to-End (E2E) Testing

Validates the full user flow — from the UI, through the API, down to the database and back. Runs in a real browser. Does the whole thing work for the user?

Type 4
Regression Testing

Re-runs existing tests after a code change to make sure nothing that was working is now broken. Did my new feature break something old?

The Testing Pyramid

Write many unit tests (fast, cheap, precise), some integration tests (medium speed, catches wiring bugs), and few E2E tests (slow, but proves the full user journey works). Most teams run unit + integration tests on every commit and E2E tests before deployment.

02
What is Jest?
The standard JavaScript testing framework — batteries included
Key Concept

Jest is a JavaScript testing framework built by Meta. It is zero-configuration, meaning it works out of the box without needing to set up a test runner, assertion library, or mocking library separately — all three are built in. It works on both the Node.js backend and React frontend of a MERN stack.

FeatureWhat it means for you
Zero config Run npx jest and it finds all *.test.js files automatically. No config file needed to get started.
Built-in assertions expect(value).toBe(), .toEqual(), .toContain() — no separate assertion library needed.
Built-in mocking jest.fn(), jest.mock() — replace real dependencies with controlled fakes without extra packages.
Code coverage Run jest --coverage to see exactly which lines of your code are tested and which aren't.
Watch mode jest --watch re-runs tests automatically every time you save a file. Instant feedback loop.
Install
Terminal 
npm install --save-dev jest

# Add to package.json scripts
# "test": "jest"
# "test:watch": "jest --watch"
# "test:coverage": "jest --coverage"
💡

If you're using ES Modules (import/export) in Node.js, add "transform": {} in jest config and use NODE_OPTIONS=--experimental-vm-modules npx jest, or switch to CommonJS (require) for test files to avoid the ESM/Jest compatibility friction.

03
Anatomy of a Jest Test
describe, test, expect — the three building blocks of every test you'll write
utils.test.js — full anatomy 
// describe() — groups related tests together
describe('calculateDiscount()', () => {

  // test() (alias: it()) — one individual test case
  test('applies 10% discount correctly', () => {
    // Arrange
    const price = 200;

    // Act
    const result = calculateDiscount(price, 10);

    // Assert
    expect(result).toBe(180);
  });

  test('throws if discount is above 100', () => {
    expect(() => calculateDiscount(200, 110)).toThrow('Invalid discount');
  });

});

Common Matchers

MatcherChecksUse for
.toBe(val) Strict equality (===) Primitives: numbers, strings, booleans
.toEqual(val) Deep equality (structure + values) Objects, arrays — use this instead of toBe
.toBeTruthy() Value is truthy Checking a value exists without caring about exact value
.toContain(item) Array includes item / string includes substring Checking arrays or error messages
.toThrow(msg) Function throws an error Testing error handling — wrap in arrow function
.toHaveBeenCalled() A mock function was called Checking side effects — did this function get triggered?
.toHaveBeenCalledWith(args) Mock called with specific arguments Checking the right data was passed to a function
.resolves / .rejects Promise resolves or rejects Async tests — await expect(promise).resolves.toBe(...)

Lifecycle Hooks

Setup and teardown 
beforeAll(async () => {
  // runs ONCE before all tests in this file
  // → connect to DB, start server, seed data
});

afterAll(async () => {
  // runs ONCE after all tests finish
  // → disconnect DB, close server
});

beforeEach(() => {
  // runs before EACH individual test
  // → reset mocks, set up fresh state
});

afterEach(async () => {
  // runs after EACH individual test
  // → clear DB collections so tests don't pollute each other
});
04
Route Testing with Supertest
Test your Express routes without actually running a server
Key Concept

Supertest lets you send real HTTP requests to your Express app in tests — without binding to a port or running a live server. It imports your Express app directly and simulates GET, POST, PUT, DELETE requests, letting you assert on status codes, response bodies, and headers.

Install
Terminal 
npm install --save-dev supertest
⚠️

Separate your app from your server. Export the Express app from app.js without calling app.listen(). Call listen() only in server.js. Supertest imports app and handles the connection itself.

app.js — export without listen 
import express from 'express';
const app = express();
app.use(express.json());
app.use('/api/users', userRoutes);

export default app; // no app.listen() here
user.test.js — route test with Supertest 
import request from 'supertest';
import app     from '../app.js';

describe('GET /api/users/:id', () => {

  test('returns user when valid ID given', async () => {
    const res = await request(app)
      .get('/api/users/123')
      .set('Authorization', 'Bearer test-token');

    expect(res.statusCode).toBe(200);
    expect(res.body).toHaveProperty('name');
    expect(res.body._id).toBe('123');
  });

  test('returns 404 when user does not exist', async () => {
    const res = await request(app)
      .get('/api/users/nonexistent');

    expect(res.statusCode).toBe(404);
    expect(res.body).toHaveProperty('error');
  });

  test('returns 401 without auth token', async () => {
    const res = await request(app)
      .get('/api/users/123');
    // no Authorization header

    expect(res.statusCode).toBe(401);
  });

});
💡

Test three things per route: the happy path (correct data → 200), the not-found case (→ 404), and the unauthorized case (no token → 401). This covers 90% of real-world bugs.

05
MongoDB Testing
Never touch your real database in tests — use mongodb-memory-server instead
Key Concept

mongodb-memory-server spins up a real MongoDB instance entirely in RAM during your test run. It behaves exactly like a real MongoDB database — you can create, save, query, and delete documents using your actual Mongoose models — but it's completely isolated from your development or production data. It auto-destructs when tests finish.

Install
Terminal 
npm install --save-dev mongodb-memory-server
Step 1 — Create a test DB helper
tests/setup/db.js 
import mongoose        from 'mongoose';
import { MongoMemoryServer } from 'mongodb-memory-server';

let mongoServer;

export const connect = async () => {
  mongoServer = await MongoMemoryServer.create();
  const uri = mongoServer.getUri();
  await mongoose.connect(uri);
};

export const disconnect = async () => {
  await mongoose.connection.dropDatabase();
  await mongoose.disconnect();
  await mongoServer.stop();
};

export const clearCollections = async () => {
  const collections = mongoose.connection.collections;
  for (const key in collections) {
    await collections[key].deleteMany({});
  }
};
Step 2 — Write model tests
user.model.test.js 
import { connect, disconnect, clearCollections } from './setup/db.js';
import User from '../models/User.js';

beforeAll(async () => await connect());
afterAll(async  () => await disconnect());
afterEach(async () => await clearCollections()); // clean slate per test

describe('User model', () => {

  test('creates and saves a user successfully', async () => {
    const user = new User({
      name:  'Harshit',
      email: 'harshit@sheryians.com',
      password: 'hashed_password_123',
    });

    const saved = await user.save();
    expect(saved._id).toBeDefined();
    expect(saved.email).toBe('harshit@sheryians.com');
  });

  test('fails validation without required email field', async () => {
    const user = new User({ name: 'NoEmail' });
    await expect(user.save()).rejects.toThrow();
  });

  test('finds user by email', async () => {
    await User.create({ name: 'Test', email: 'find@test.com', password: 'pass' });

    const found = await User.findOne({ email: 'find@test.com' });
    expect(found).toBeTruthy();
    expect(found.name).toBe('Test');
  });

});
💡

Why afterEach(clearCollections) matters: Without it, a user created in test 1 is still in the DB when test 2 runs. Tests start depending on each other's data. That's how you get tests that pass individually but fail when run together. Always clean up.

06
Mocking with Jest
Replace real dependencies with controlled fakes so tests are fast and predictable
Key Concept

A mock is a fake version of a dependency (a function, a module, an API call) that returns whatever you tell it to. You use mocks when the real dependency is slow (a database), has side effects (sending an email), or is external (a third-party API). Mocks make tests fast, deterministic, and isolated.

jest.fn() — Mock a single function

Mocking a service function 
const sendEmail = jest.fn(); // fake function — does nothing by default

sendEmail('user@example.com');

expect(sendEmail).toHaveBeenCalledTimes(1);
expect(sendEmail).toHaveBeenCalledWith('user@example.com');

// Make it return a value
const getUser = jest.fn().mockResolvedValue({ name: 'Harshit', id: 1 });
const user = await getUser(1);
expect(user.name).toBe('Harshit');

jest.mock() — Mock an entire module

Mocking a DB service module 
// Mock the whole userService module
jest.mock('../services/userService.js', () => ({
  findUserById: jest.fn().mockResolvedValue({
    _id: '123',
    name: 'Harshit',
    email: 'harshit@sheryians.com'
  }),
  deleteUser: jest.fn().mockResolvedValue({ deleted: true }),
}));

// Now any test that imports userService gets the fake version
// Real DB is never touched — tests run instantly

Spy on a real function without replacing it

jest.spyOn() 
import * as emailService from '../services/emailService.js';

test('sends welcome email on register', async () => {
  // Watch the real function but stop it from actually sending
  const spy = jest.spyOn(emailService, 'sendWelcome')
    .mockResolvedValue(true);

  await request(app).post('/api/auth/register').send({...});

  expect(spy).toHaveBeenCalledTimes(1);
  spy.mockRestore(); // restore original after test
});
07
Frontend Testing Landscape
What the industry uses in 2025 — and why Jest alone isn't enough for React

The frontend testing world has moved on from plain Jest. Modern React projects — especially Vite-based ones — use a different stack. Here's what each tool is, why it exists, and when you'd use it.

Vitest
Jest replacement

The modern replacement for Jest in Vite-based React projects. Natively runs inside Vite — no separate build step, no config fighting. Jest-compatible API so migration is near-zero effort.

  • Significantly faster than Jest in Vite projects (shares the same transform pipeline)
  • TypeScript support out of the box — no Babel config needed
  • Hot Module Replacement in watch mode — tests re-run in milliseconds as you type
  • Same API as Jest: describe, test, expect, vi.mock()
Use for: unit tests, hook tests, utility function tests in React + Vite projects
React Testing Library
Component testing

The standard for testing React components. Its core philosophy: test components the way a real user would — by clicking buttons, typing in inputs, and reading visible text — not by poking at internal state or class names.

  • Queries by accessibility: getByRole, getByLabelText, getByText
  • Forces you to write accessible components as a side effect of writing testable ones
  • Works with both Jest and Vitest — just change the test runner, RTL stays the same
  • userEvent simulates real typing, clicking, and form interactions (better than fireEvent)
Use for: component behavior tests — does the button work, does the form validate, does the modal open
Playwright
E2E testing

The top choice for end-to-end testing in 2025. Runs tests in real browsers (Chromium, Firefox, WebKit). Built-in auto-waiting eliminates the flaky timing failures that plagued older E2E tools.

  • codegen — records your browser clicks and auto-generates test code
  • Runs tests in parallel across multiple browsers simultaneously
  • No manual waitFor calls — Playwright waits for elements automatically
  • Free, open source, Microsoft-backed — no paid tier needed for most teams
Use for: full user flow tests — login → dashboard → create item → verify it appears
Cypress
E2E alternative

A popular E2E testing tool known for its excellent developer experience and visual test runner UI. Great for teams new to E2E testing — the time-travel debugger makes failures easy to understand.

  • Visual test runner shows the browser in real time as tests run
  • Excellent error messages and time-travel debugging
  • Slower than Playwright — tests run sequentially by default
  • Advanced dashboard features (test analytics, parallelization) require a paid plan
Use for: teams that prioritize DX and visual debugging over raw speed
Storybook
Visual testing

A tool for building and visually inspecting components in complete isolation from the rest of the app. Not a test runner — it's a component workbench. You see exactly how a component looks in every state.

  • Build Button, Modal, Card in isolation — no need to navigate the full app
  • Catches visual regressions — a screenshot diff tells you if a button's padding changed
  • Living documentation — designers and developers share the same source of truth
Use for: design systems, component libraries, visual appearance testing

When to Use What
The decision map — one question, one tool
What you want to testReach forWhy
A utility / helper function Jest / Vitest Pure logic, no DOM, no browser needed
An Express route / API endpoint Jest + Supertest Simulates HTTP without a real running server
A Mongoose model / DB operation Jest + mongodb-memory-server Real MongoDB behavior, zero risk to real data
A React component's behavior Vitest + React Testing Library Tests what the user sees and does, not internals
A full user flow (login → dashboard) Playwright Real browser, real navigation, end-to-end proof
What a component looks like visually Storybook Isolated rendering, screenshot comparison

The recommended 2025 frontend stack: Vitest for unit + component tests (fast, Vite-native), React Testing Library for component behavior (user-centric), Playwright for E2E (real browser, auto-waiting, free). This combination covers every layer of your frontend with the right tool.
