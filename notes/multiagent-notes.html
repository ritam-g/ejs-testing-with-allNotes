01
The Problem with a Single Agent
Why one LLM doing everything breaks down in production

Imagine asking one person to be your lawyer, accountant, doctor, and software engineer — all at the same time, in the same conversation. They'll give you average answers on everything rather than expert answers on anything. LLMs have the exact same problem.

🧠
Context Window Overflow
Every LLM has a maximum context window — typically 8K to 128K tokens. Complex tasks (research a topic, write code, review it, then deploy) quickly overflow this limit. The model starts "forgetting" earlier parts of the conversation.
Fix → split tasks across agents, each with a fresh context
👻
Hallucination Amplification
When a single agent handles the entire pipeline — research, reasoning, and output — one hallucination early in the chain poisons every downstream step. There's no checker, no verifier, no second opinion.
Fix → reviewer agent independently verifies outputs
🐌
Slow Sequential Processing
A single agent must do everything one after another. Research → then summarise → then write code → then test. Every step waits for the previous. With complex tasks this means very long wait times.
Fix → parallel agents run simultaneously
🎯
Jack of All Trades, Master of None
One agent given a massive system prompt trying to be a researcher, coder, writer, and analyst simultaneously produces mediocre results in all areas. Its attention and "focus" is diluted across too many roles.
Fix → specialist agents with focused prompts
💥
Single Point of Failure
If one step fails in a single-agent pipeline, the entire task fails. There's no fallback, no retry mechanism, no isolation. One bad tool call crashes the whole workflow.
Fix → isolated agents fail independently
🔧
Unmaintainable Prompts
As you try to make one agent do more things, its system prompt grows into a massive 2000-word instruction set. Changing one behaviour breaks another. It becomes impossible to test or improve.
Fix → small focused prompts per agent, easy to tune
Single agent handling a "Build a product feature" task:
 → Research market (uses 20K tokens)
 → Write spec (uses 15K tokens) — already forgetting early research
 → Write code (uses 30K tokens) — context limit hit, starts hallucinating
 → Review code — forgets the spec, reviews against wrong criteria
 Result: inconsistent, hallucinated, slow ✗

Multiagent system:
 Research Agent → focused, full context, accurate research
 Spec Agent → reads research output, writes tight spec
 Code Agent → given spec only, writes clean code
 Review Agent → independently checks code against spec
 Result: accurate, fast, maintainable ✅
02
What is a Multiagent System?
Specialised agents that communicate — each an expert, together unstoppable
Key Concept

A multiagent system is a collection of AI agents where each agent has a specific role, its own system prompt, its own tools, and its own context window. A primary (orchestrator) agent receives the user's task, breaks it into subtasks, and delegates to sub-agents as tools. Each sub-agent does one thing extremely well and returns its result to the orchestrator, which assembles the final answer.

Single Agent vs Multiagent

FactorSingle AgentMultiagent System
Context managementOne giant context that overflows on complex tasksEach agent has its own focused context ✓
Hallucination riskHigh — no verification stepLower — reviewer agent catches errors ✓
SpeedSequential — one thing at a timeParallel agents can run simultaneously ✓
SpecialisationMediocre at everythingEach agent is expert at one thing ✓
Failure isolationOne failure = total failureAgents fail independently, retries possible ✓
MaintainabilityOne giant prompt — hard to debugSmall focused prompts — easy to tune ✓
💡

Real-world analogy: A multiagent system is like a company. The CEO (primary agent) receives a goal and delegates to specialists — the legal team, the engineering team, the marketing team (sub-agents). Each team is expert in their domain and reports results back. The CEO assembles the final deliverable.

04
Design Patterns
The four ways multiagent systems are structured — pick the right one
Pattern 1
Orchestrator → Workers

One primary agent delegates to specialist sub-agents as tools. Sub-agents report back, primary assembles the answer. This is what we're building. Best for: complex multi-step tasks.

Pattern 2
Pipeline (Chain)

Agents run in sequence — Agent A's output feeds Agent B's input. No orchestrator needed. Best for: linear workflows like research → draft → review → publish.

Pattern 3
Parallel Workers

Multiple agents run the same task simultaneously with different approaches. Results are compared or merged. Best for: consensus, fact-checking, reducing hallucination risk.

Pattern 4
Debate / Critic

One agent generates output, another agent critiques it, the first revises. Repeat until quality threshold met. Best for: writing, code review, argument quality.

Mistral models at a glance

ModelBest forTool calling
mistral-large-latest Complex reasoning, orchestrator agent — most capable ✅ Yes
mistral-small-latest Fast sub-agents, simple tasks, lower cost ✅ Yes
codestral-latest Specialist code generation sub-agent — purpose-built for code ✅ Yes
mistral-tiny Cheapest — simple classification tasks only ❌ No
⚠️

Tool calling requires a capable model. Not all Mistral models support tool/function calling. Always use mistral-large-latest, mistral-small-latest, or codestral-latest for agents. Using mistral-tiny with tools will throw a 422 extra_forbidden error from Mistral's API.

Core concepts in LangChain agents

ConceptWhat it isIn our system
createAgent The modern high-level agent builder from langchain. Wraps LangGraph internally. No AgentExecutor or ChatPromptTemplate needed — just pass model, tools, and a system prompt string. Used to create all 4 agents (primary + 3 sub)
tool() The modern way to define a tool — takes a function and a config object with name, description, and Zod schema. Also used to wrap sub-agents so the primary agent can call them. Used to create all tools AND wrap sub-agents
agent.invoke() Runs the agent's reasoning loop directly. Takes { messages: [...] } — no separate AgentExecutor needed. The loop runs until the model emits a final answer or hits the iteration limit. Called on each sub-agent executor and the primary agent
systemPrompt A plain string passed directly to createAgent. Replaces the verbose ChatPromptTemplate.fromMessages([['system', ...]]) boilerplate from the old pattern. Defines each agent's personality and role
z (Zod) Schema library used to describe tool input shapes. LangChain uses Zod schemas to validate and document tool inputs for the LLM. Defines the input schema for every tool
06
Building the Three Sub-Agents
Each agent created with createAgent — then wrapped with tool()
The Pattern (createAgent)

Every sub-agent follows the same 3-step pattern: (1) Create it with createAgent from langchain — pass a systemPrompt string, the model, and its specific tools. No ChatPromptTemplate, no agent_scratchpad, no AgentExecutor needed. (2) Call agent.invoke({ messages: [...] }) directly to run it. (3) Wrap the whole thing in a tool() with a Zod schema so the primary agent can call it like any other tool.

Research Agent

src/agents/researchAgent.js
import { createAgent }       from 'langchain';               // ← top-level langchain package
import { tool }              from '@langchain/core/tools';
import { ChatMistralAI }     from '@langchain/mistralai';
import { z }                 from 'zod';
import 'dotenv/config';

const llm = new ChatMistralAI({ model: 'mistral-large-latest', temperature: 0.3 });

// Research agent's own internal tool
const webSearchTool = tool(
  async ({ query }) => {
    // Replace with a real search API (Tavily, SerpAPI, Brave, etc.)
    return `Search results for "${query}": [simulated results here]`;
  },
  {
    name:        'web_search',
    description: 'Search the internet for current information on a topic',
    schema: z.object({
      query: z.string().describe('The search query to look up'),
    }),
  }
);

// createAgent — no ChatPromptTemplate, no AgentExecutor needed
const researchAgent = createAgent({
  model:        llm,
  tools:        [webSearchTool],
  systemPrompt: `You are a specialist research agent. Your only job is to
gather accurate, relevant information on a given topic.
Use the web_search tool to find information.
Return a concise, factual summary. Do not write code.
Do not give opinions. Only report what you find.`,
});

// Wrap the agent as a tool for the primary orchestrator
export const researchTool = tool(
  async ({ topic }) => {
    const result = await researchAgent.invoke({
      messages: [{ role: 'user', content: topic }],
    });
    // result.messages — grab the last message (the agent's final answer)
    const last = result.messages.at(-1);
    return last?.content ?? '';
  },
  {
    name:        'research_agent',
    description: 'Use this to research any topic. Returns a factual summary. Input: the topic or question to research.',
    schema: z.object({
      topic: z.string().describe('The topic or question to research'),
    }),
  }
);

Code Agent codestral-latest

src/agents/codeAgent.js
import { createAgent }   from 'langchain';
import { tool }          from '@langchain/core/tools';
import { ChatMistralAI } from '@langchain/mistralai';
import { z }             from 'zod';
import 'dotenv/config';

// Code agent uses Codestral — Mistral's purpose-built coding model
const codeLLM = new ChatMistralAI({
  model:       'codestral-latest',  // specialist model just for code
  temperature: 0.1,                 // very low — code must be precise
});

const codeAgent = createAgent({
  model:        codeLLM,
  tools:        [],        // code agent needs no external tools
  systemPrompt: `You are a specialist code writing agent. Your only job
is to write clean, working JavaScript/Node.js code based on
the specification given to you.
Write complete, runnable code with comments.
Do not do research. Do not review. Only write code.`,
});

export const codeTool = tool(
  async ({ specification }) => {
    const result = await codeAgent.invoke({
      messages: [{ role: 'user', content: specification }],
    });
    const last = result.messages.at(-1);
    return last?.content ?? '';
  },
  {
    name:        'code_agent',
    description: 'Use this to write code. Input: a clear description of what the code should do. Returns working JavaScript/Node.js code.',
    schema: z.object({
      specification: z.string().describe('A clear description of the code to write'),
    }),
  }
);

Review Agent

src/agents/reviewAgent.js
import { createAgent }   from 'langchain';
import { tool }          from '@langchain/core/tools';
import { ChatMistralAI } from '@langchain/mistralai';
import { z }             from 'zod';
import 'dotenv/config';

const llm = new ChatMistralAI({ model: 'mistral-large-latest', temperature: 0.3 });

const reviewAgent = createAgent({
  model:        llm,
  tools:        [],
  systemPrompt: `You are a specialist review and quality-assurance agent.
Your only job is to critically review content or code given to you.
Check for: accuracy, completeness, logical errors, hallucinations,
security issues (for code), and missing edge cases.
Be specific about what is wrong and what needs to change.
Do not write new content. Only review and give feedback.`,
});

export const reviewTool = tool(
  async ({ content }) => {
    const result = await reviewAgent.invoke({
      messages: [{ role: 'user', content: content }],
    });
    const last = result.messages.at(-1);
    return last?.content ?? '';
  },
  {
    name:        'review_agent',
    description: 'Use this to review and validate any content or code for quality, accuracy, and correctness. Input: the content to review. Returns detailed feedback.',
    schema: z.object({
      content: z.string().describe('The content or code to review and validate'),
    }),
  }
);
💡

What changed vs createReactAgent? Three things are now gone: ChatPromptTemplate.fromMessages(), the agent_scratchpad placeholder, and AgentExecutor. With createAgent you write a plain systemPrompt string and call agent.invoke({ messages: [...] }) directly. Also note the response shape: createAgent returns { messages: [...] } instead of { output: '...' }, so we extract the final answer with result.messages.at(-1)?.content.

07
The Primary (Orchestrator) Agent
The agent that takes the user's task and decides which sub-agents to call
src/agents/primaryAgent.js
import { createReactAgent, AgentExecutor } from 'langchain/agents';
import { ChatPromptTemplate }              from '@langchain/core/prompts';
import { llm }                             from '../llm.js';

// Import sub-agents — each is already wrapped as a tool() export
import { researchTool } from './researchAgent.js';
import { codeTool }     from './codeAgent.js';
import { reviewTool }   from './reviewAgent.js';

// All sub-agents registered as tools on the orchestrator
const tools = [researchTool, codeTool, reviewTool];

// Orchestrator prompt — high-level planning, not execution
const primaryPrompt = ChatPromptTemplate.fromMessages([
  ['system', `You are the primary orchestrator agent. You receive complex
tasks from the user and coordinate specialist sub-agents to complete them.

You have three tools:
- research_agent: use when you need facts, information, or data
- code_agent: use when you need to write or generate code
- review_agent: use when you need to verify, check, or validate content

Your job is to:
1. Break the task into subtasks
2. Delegate each subtask to the right specialist agent
3. Use the review_agent to verify important outputs
4. Assemble and return a complete, high-quality final answer

Always review code before returning it. Always verify facts when accuracy matters.`],
  ['human', '{input}'],
  ['placeholder', '{agent_scratchpad}'],
]);

const primaryAgent = await createReactAgent({
  llm,          // mistral-large-latest from llm.js
  tools,
  prompt: primaryPrompt,
});

export const primaryExecutor = new AgentExecutor({
  agent:                   primaryAgent,
  tools,
  verbose:                 true,   // log each step so you can see delegation
  maxIterations:           10,    // prevent infinite loops
  returnIntermediateSteps: true,  // expose which agent was called + outputs
});
⚠️

Always set maxIterations. Without it, a confused agent can loop forever, calling tools repeatedly and burning API credits. 10 iterations is a safe limit for most tasks — the agent will stop and return whatever it has if it hits the limit.

Full Working Example
Entry point, folder structure, and what the agent trace looks like
Entry point
src/index.js
import { primaryExecutor } from './agents/primaryAgent.js';

const task = `
  Research what a REST API is, then write a simple Express.js
  REST API with GET and POST routes for a todos list,
  then review the code for any issues.
`;

const result = await primaryExecutor.invoke({ input: task });

console.log('\n=== FINAL ANSWER ===');
console.log(result.output);

// See every step — which sub-agent was called and what it returned
console.log('\n=== AGENT STEPS ===');
result.intermediateSteps.forEach((step, i) => {
  console.log(`Step ${i + 1}: Called → ${step.action.tool}`);
  console.log(`  Input:  ${JSON.stringify(step.action.toolInput)}`);
  console.log(`  Output: ${String(step.observation).slice(0, 120)}...`);
});

Key takeaways

What we builtWhy it matters
ChatMistralAI Drop-in replacement for ChatOpenAI. Reads MISTRAL_API_KEY from env. Use mistral-large-latest for orchestration and codestral-latest for the code specialist.
tool() + Zod schema Modern replacement for DynamicTool. Zod schemas give Mistral's function-calling precise knowledge of parameter names and types — far more reliable than string-only descriptions.
Sub-agents as tools The primary agent sees sub-agents exactly like any other tool. This is the key design insight — agents become infinitely composable with zero extra framework.
Automatic delegation Mistral's function-calling decides when and which sub-agent to call based on the task. You never hardcode the workflow — the LLM plans it.
maxIterations safety Prevents infinite agent loops. Production systems always set this. Without it you'll burn API budget on a stuck agent that never returns.
The Big Picture

This pattern is exactly how production AI products work today. Cursor uses a code agent, a reviewer agent, and a planner. Devin is a multiagent system with specialised sub-agents. By switching from OpenAI to Mistral here, you're using one of the most capable open-weights model families available — with function calling, competitive pricing, and a model purpose-built for code (Codestral). The future of software development is writing agents that write software — and understanding how to architect them well is one of the most valuable skills you can have right now.
