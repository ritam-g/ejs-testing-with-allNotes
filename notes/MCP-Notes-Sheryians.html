01
Life Before MCP
The painful problems MCP was built to solve

Before MCP existed, connecting an LLM to external tools was a custom engineering job — every single time. Every model, every tool, every company wrote their own glue code from scratch. Here's what that looked like.

🔌
N × M Integration Hell
If you had 5 LLMs and 10 tools (Slack, GitHub, database, file system, calendar...) you needed 50 custom integrations. Each one was bespoke — different auth, different format, different error handling. Change one tool and you break integrations with every model.
MCP fix → one standard server, works with any MCP-compatible LLM
📋
Copy-Pasting Context Manually
Want Claude to read your codebase? You copy-paste files into the chat. Want it to reference your docs? Copy-paste again. Want it to check the database? Manually query and paste the results in. There was no way for the AI to reach out and get context itself.
MCP fix → Resources let AI pull context on demand
🔄
Reinventing the Wheel Every Time
Teams at different companies all built GitHub integrations for their AI tools independently. All different formats, all incompatible, all separately maintained. The same work done thousands of times across the industry with no shared standard.
MCP fix → build once, publish to registry, anyone uses it
🔒
No Standard Security Model
Every custom integration handled auth differently. Some passed API keys in plaintext. Some had no sandboxing. There was no consistent model for what an AI was and wasn't allowed to do with a connected tool.
MCP fix → defined auth model (OAuth 2.1), sandboxing, consent flows
🧩
Tools Locked to One AI
A tool you built for ChatGPT didn't work with Claude. A plugin for Claude didn't work with Gemini. Every AI platform had its own proprietary plugin format. You couldn't reuse anything across models.
MCP fix → model-agnostic standard — same server works with Claude, GPT, Gemini
⚡
No Discovery Mechanism
The LLM had no way to ask "what tools are available?" You had to hardcode every capability into the system prompt upfront. Adding a new tool meant updating every prompt in every deployment.
MCP fix → dynamic tool discovery via tools/list at runtime
02
What is MCP?
An open standard — not a framework, not a library, a protocol
Key Concept

MCP (Model Context Protocol) is an open standard published by Anthropic in November 2024 that defines how AI applications discover and invoke external tools, read data, and use reusable prompt templates — all through a consistent JSON-RPC interface. Think of it like HTTP for AI integrations: it doesn't matter which LLM or which tool — if both speak MCP, they work together.

The USB-C analogy:
 Before USB-C: every device had a different cable. Nokia charger ≠ Motorola charger.
 USB-C: one standard port, every device works with every charger.

Before MCP: every AI needed custom code to talk to every tool.
After MCP: one MCP server, every MCP-compatible AI talks to it. ✅

Who has adopted MCP (as of 2025)

AI Providers
Adopted MCP natively

Anthropic (Claude), OpenAI (ChatGPT), Google DeepMind (Gemini), Microsoft (Copilot), Cloudflare — all officially support the MCP standard.

Developer Tools
Built-in MCP support

Cursor IDE, VS Code (GitHub Copilot), Claude Desktop, Claude Code — all natively connect to MCP servers. You can plug in any MCP server and they instantly gain its capabilities.

How it works — the basic flow

StepWhat happensWho does it
1. ConnectMCP client connects to the server, negotiates protocol version and capabilitiesClient (Claude, Cursor, etc.)
2. DiscoverClient calls tools/list, resources/list to see what's availableClient asks server
3. InjectTool definitions injected into LLM's context window as available capabilitiesClient / host
4. DecideLLM decides which tool to call based on the user's requestLLM
5. CallClient sends tools/call request to MCP server with the arguments LLM choseClient executes
6. ReturnServer runs the tool, returns the result back to the LLM's contextServer → LLM

Key insight: The MCP Client and the LLM both live inside the Host application (like Claude Desktop). The MCP Server is your code — a separate process that the host spawns and communicates with over stdin/stdout (locally) or HTTP+SSE (remotely). 04
MCP Primitives
Three types of things an MCP server can expose — Tools, Resources, and Prompts

Every MCP server exposes one or more of three primitives. Each has a different purpose and a different interaction model. Together they cover every way an AI needs to interact with external systems.

🔧 Tools
Model-controlled · LLM decides when to call

Tools are executable functions. The LLM autonomously decides to call a tool when it determines it needs to. Tools have a name, description, and JSON Schema input definition. Think of them like function calls — the LLM passes arguments, the server runs the function, and returns a result.

  • Examples: send_email, query_database, create_github_pr, get_weather
  • The LLM reads the tool description and decides when calling it is appropriate
  • Can have side effects — they actually do things, not just read things
  • Discovered via tools/list — server returns all available tool definitions
  • Called via tools/call — server executes the function and returns output
📄 Resources
Application-controlled · host app exposes to LLM

Resources are read-only data that the server exposes for context. Unlike tools (which do things), resources are data sources the LLM can read from — files, database rows, config, documentation. Addressed by URI like file:///config.json or db://users/42.

  • Examples: file:///src/app.js, db://schema/users, config://settings
  • Read-only — no side effects, just data access
  • Discovered via resources/list — server returns a list of available URIs
  • Read via resources/read — returns the content at that URI
  • Use for: codebase context, documentation, database schemas, configuration
💬 Prompts
User-controlled · reusable workflow templates

Prompts are pre-written message templates the user can select to start a structured workflow. They're like slash commands — /debug-this-error or /write-tests-for. The server defines the template and what arguments it takes; the user picks it and fills in the arguments.

  • Examples: /code-review, /explain-error, /write-migration
  • Can embed resources directly — e.g. auto-fetch the relevant file into the prompt
  • Discovered via prompts/list and retrieved via prompts/get
  • Different from Tools: the user explicitly triggers a Prompt; the LLM autonomously triggers a Tool
  • Underused feature — most servers only expose Tools, but Prompts enable powerful workflow automation

Quick comparison

PrimitiveWho triggers itSide effectsBest for
ToolLLM decides autonomouslyYes — it does thingsActions: query DB, send email, run code
ResourceApplication / host exposesNo — read onlyContext: files, docs, schemas, config
PromptUser explicitly picksNo — template onlyWorkflows: slash commands, structured tasks

Setting Up an MCP Server
Install the SDK, create the server, connect the transport
Step 1 — Install
Terminal
npm init -y
npm install @modelcontextprotocol/sdk

# Add to package.json:
# "type": "module"
# "bin": { "my-server": "./src/index.js" }
Step 2 — Create the server skeleton
src/index.js
#!/usr/bin/env node
import { McpServer }           from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';

// Create the server — give it a name and version
const server = new McpServer({
  name:    'my-mcp-server',
  version: '1.0.0',
});

// ── Register tools, resources, prompts here ──────
// (see sections 6 and 7)

// Connect via stdio transport (local usage)
const transport = new StdioServerTransport();
await server.connect(transport);

Transport types

TransportHow it worksUse for
stdio Host spawns the server as a child process. JSON-RPC messages sent over stdin/stdout. No network involved. Local tools — Cursor, Claude Desktop, Claude Code. Default for most servers.
HTTP + SSE Server runs as an HTTP server. Clients POST requests; server streams responses via Server-Sent Events. Remote servers, multi-client deployments, cloud-hosted MCP servers.
💡

The shebang line (#!/usr/bin/env node) at the top of index.js makes the file directly executable on Unix systems. This is required for Claude Desktop to spawn your server process correctly.

06
Implementing Tools
Register functions the LLM can call — with name, description, and schema
Key Concept

A tool definition has three parts: name (what the LLM calls it), description (how the LLM decides when to use it — write this well!), and inputSchema (the JSON Schema defining what arguments it accepts). The LLM reads all three and autonomously decides when and how to call your tool.

Basic tool — no arguments
src/index.js
import { z } from 'zod'; // for schema validation

// Tool with no required arguments
server.tool(
  'get_server_time',
  'Returns the current server date and time in ISO format',
  {},                           // empty schema — no args needed
  async () => ({
    content: [{
      type:  'text',
      text:  new Date().toISOString(),
    }]
  })
);
Tool with arguments — Zod schema
Tool with input schema
// npm install zod
import { z } from 'zod';

server.tool(
  'add_numbers',
  'Adds two numbers together and returns the result',
  {
    a: z.number().describe('First number'),
    b: z.number().describe('Second number'),
  },
  async ({ a, b }) => ({
    content: [{
      type: 'text',
      text: `${a} + ${b} = ${a + b}`,
    }]
  })
);
Real-world tool — query a database
Database query tool
server.tool(
  'query_users',
  'Search for users in the database by name or email. Returns matching user records.',
  {
    search: z.string().describe('Name or email to search for'),
    limit:  z.number().optional().default(10).describe('Max results to return'),
  },
  async ({ search, limit }) => {
    try {
      const users = await db.query(
        `SELECT id, name, email FROM users
         WHERE name ILIKE $1 OR email ILIKE $1
         LIMIT $2`,
        [`%${search}%`, limit]
      );
      return {
        content: [{
          type: 'text',
          text: JSON.stringify(users.rows, null, 2),
        }]
      };
    } catch (err) {
      return {
        content: [{ type: 'text', text: `Error: ${err.message}` }],
        isError: true,
      };
    }
  }
);
⚠️

The description is a prompt — write it carefully. The LLM reads your tool description to decide when to call it. Vague descriptions like "does stuff with users" lead to wrong tool selection. Be specific: what it does, what it returns, when to use it. Bad description = bad tool calls.

07
Implementing Resources
Expose read-only data the LLM can pull as context
src/index.js — resources
import { ListResourcesRequestSchema, ReadResourceRequestSchema }
  from '@modelcontextprotocol/sdk/types.js';
import fs from 'fs/promises';

// ── LIST — tell clients what resources exist ──────
server.setRequestHandler(ListResourcesRequestSchema, async () => ({
  resources: [
    {
      uri:      'file:///config/settings.json',
      name:     'App Settings',
      mimeType: 'application/json',
    },
    {
      uri:      'db://schema/users',
      name:     'Users table schema',
      mimeType: 'text/plain',
    },
  ]
}));

// ── READ — return the content for a given URI ─────
server.setRequestHandler(ReadResourceRequestSchema, async (req) => {
  const { uri } = req.params;

  if (uri === 'file:///config/settings.json') {
    const content = await fs.readFile('./config/settings.json', 'utf-8');
    return {
      contents: [{ uri, mimeType: 'application/json', text: content }]
    };
  }

  if (uri === 'db://schema/users') {
    return {
      contents: [{
        uri,
        mimeType: 'text/plain',
        text: `CREATE TABLE users (
  id      SERIAL PRIMARY KEY,
  name    VARCHAR(255) NOT NULL,
  email   VARCHAR(255) UNIQUE NOT NULL,
  created TIMESTAMPTZ DEFAULT NOW()
);`
      }]
    };
  }

  throw new Error(`Unknown resource URI: ${uri}`);
});
💡

Resources vs Tools for data access: Use a Resource when the data is static or semi-static context (a file, a schema, a config). Use a Tool when fetching data requires dynamic arguments (search for user by ID, query by date range). Resources are indexed and listable; Tools are called with specific inputs.

Build an Agent with LangChain + MistralAI
Connect your MCP server to a Mistral-powered agent in JavaScript

So far you've built an MCP server. Now let's build an agent that actually uses it. Instead of wiring it to Claude Desktop (a GUI), we'll write a JS script that connects to our server, loads its tools, and lets Mistral decide when to call them.

The idea in one line

@langchain/mcp-adapters reads your MCP server's tools and turns them into LangChain tools. Then createReactAgent gives those tools to Mistral. The model decides when to call them — you just ask a question and get an answer.

Step 1 — Install dependencies
Terminal
npm install @langchain/mcp-adapters @langchain/mistralai @langchain/langgraph @langchain/core dotenv
Step 2 — Add your Mistral API key
.env
# Get a free key at console.mistral.ai
MISTRAL_API_KEY=your-api-key-here
Step 3 — Create the agent
agent.js
import { MultiServerMCPClient } from '@langchain/mcp-adapters';
import { ChatMistralAI }        from '@langchain/mistralai';
import { createReactAgent }     from '@langchain/langgraph/prebuilt';
import { HumanMessage }         from '@langchain/core/messages';
import 'dotenv/config';

// 1. Connect to your MCP server
const client = new MultiServerMCPClient({
  mcpServers: {
    myServer: {
      transport: 'stdio',
      command:   'node',
      args:      ['./src/index.js'], // path to your MCP server
    },
  },
});

// 2. Load the tools from your server
const tools = await client.getTools();

// 3. Set up the Mistral model
const model = new ChatMistralAI({
  model:       'mistral-large-latest',
  temperature: 0,
});

// 4. Create the agent — give it the model and tools
const agent = createReactAgent({ llm: model, tools });

// 5. Ask a question — the agent calls your tools automatically
const result = await agent.invoke({
  messages: [new HumanMessage('What is the current server time?')],
});

console.log(result.messages.at(-1).content);

// 6. Clean up
await client.close();
Step 4 — Run it
Terminal
node agent.js

That's it. The agent reads your MCP server's tools, passes them to Mistral, and Mistral figures out which one to call — you don't wire anything manually.

💡

What's happening behind the scenes: getTools() calls tools/list on your MCP server and converts every tool into a LangChain tool object. createReactAgent runs a loop — Mistral sees the tools, decides to call one, gets the result back, and decides to answer or call another. You never write that loop yourself.

⚠️

Always call client.close() at the end. The MCP client spawns your server as a child process. If you skip close(), that process keeps running in the background even after your script finishes.

The Problems MCP Creates
MCP solves the integration problem — but introduces a new one: tool bloat

MCP is a massive improvement over custom integrations. But as teams connect more and more MCP servers, a new set of problems emerges. Understanding these is part of using MCP professionally.

Problem 1 — Tool Definition Bloat in the Context Window

The Core Problem

Every MCP tool connected to your AI agent has its full schema injected into the LLM's context window on every single request — its name, description, parameter types, and descriptions. The LLM needs to see all of this to know what's available. But context windows are finite, and tool schemas are expensive.

Token cost of tool metadata — real numbers
Playwright MCP server (21 tools)
~11,700 tokens
Claude's built-in tools
~11,600 tokens
5 servers × 30 tools each
30,000–60,000 tokens
Mistral large context window
128,000 tokens

5 MCP servers with 30 tools each can consume 25–50% of Mistral's context window before the user types their first message.

Problem 2 — Wrong Tool Selection at Scale

When a dozen tools are in context, the LLM sometimes picks the wrong one. If two tools have similar descriptions — search_users and find_contacts — the model gets confused. The more tools injected, the higher the interference between them, and the less reliably the LLM chooses the right one.

5 MCP servers connected simultaneously:
  Slack server: 12 tools · GitHub server: 18 tools
  Postgres server: 8 tools · Notion server: 14 tools · Your server: 6 tools
  = 58 tool definitions injected — ~25,000 tokens — before any user message

  LLM calls create_slack_message when it should call create_github_issue
  Tool descriptions overlap → model picks wrong one → task fails

Problem 3 — Higher API Cost Per Request

Every API call to Mistral charges you per token. Tool definitions are injected into every single request — not just the ones where tools are used. A simple "what time is it?" question still injects all 58 tool schemas into the context. You pay for all those tokens every time.

Problem 4 — Security Surface Area

Every connected MCP server is a potential attack surface. MCP tools can execute code, read files, query databases, and send messages. A compromised or malicious MCP server could do real damage. The more servers you connect, the larger your attack surface. This is an active concern in the industry as of 2025.

How the industry handles these problems

ProblemCurrent best practice
Tool bloat Build focused MCP servers. Don't create one server for everything. One server per domain (one for GitHub, one for DB). Keep tool count under 10–15 per server.
Wrong tool selection Write extremely clear, distinct descriptions. Name tools unambiguously. In MultiServerMCPClient, use prefixToolNameWithServerName: true to avoid cross-server name collisions.
Token cost Only connect the MCP servers actually needed for a given agent. Lazy-load tool schemas. Use mistral-small-latest for simpler tasks to save cost while keeping tool-calling capability.
Security Vet community servers before running them. Sandbox server processes. Apply principle of least privilege — only expose the capabilities you actually need.
The Mental Model — MCP in 2025

MCP solved the integration problem — you no longer need custom glue code for every model-tool pair. It is now the industry standard, adopted by every major AI provider. But it introduced the scale problem — the more tools you connect, the worse the LLM performs. The skill in using MCP professionally is knowing what not to connect. Small, focused servers with fewer than 15 well-described tools consistently outperform one giant server with 100 tools.
