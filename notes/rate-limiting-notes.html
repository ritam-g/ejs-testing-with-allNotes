01
What is Rate Limiting?
The bouncer at the door — legitimate users never notice it, bots get stopped cold
Key Concept

Rate limiting controls how many requests a client can make to your server within a defined time window. A client that exceeds the limit gets a 429 Too Many Requests response. Legitimate users, who make maybe 10–20 requests per minute, never hit the limit. Bots and attackers making thousands of requests per second are blocked.

What it Protects Against

Protection
Brute-force attacks

Attackers trying thousands of password combinations on /login. Rate limiting stops them after 5–10 attempts.

Protection
Server overload

A single buggy client or scraped loop hammering your API. Rate limiting caps the damage from one bad actor.

Protection
API abuse / scraping

Bots scraping your entire product catalogue in seconds. Rate limiting slows them to a pace your server handles easily.

Protection
Cloud cost spikes

More requests = more compute, more DB reads, higher bills. Rate limiting puts a hard ceiling on resource consumption per client.

Without rate limiting:
  Bot sends 10,000 requests/min → DB overwhelmed → server crashes → all users affected

With rate limiting:
  Bot sends 10,000 requests/min
    ↓
After request #100 → 429 Too Many Requests ← bot is stopped
Real users (10 req/min) → 200 OK ← unaffected ✅
02
express-rate-limit Setup
Install, configure globally, and apply selectively to sensitive routes
Step 1 — Install
Terminal 
npm install express-rate-limit
Step 2 — Global limiter (all routes)
app.js 
import rateLimit from 'express-rate-limit';

const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 100,                    // 100 requests per window per IP
  message: {
    error: 'Too many requests. Please try again later.'
  },
  statusCode: 429,
  standardHeaders: true,   // sends RateLimit-* headers
  legacyHeaders: false,
});

// Apply to every route
app.use(globalLimiter);
Step 3 — Stricter limiter on sensitive routes
auth routes 
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 10,                     // only 10 attempts per window
  message: { error: 'Too many login attempts. Try again in 15 minutes.' },
  skipSuccessfulRequests: true, // don't count successful logins
});

// Apply only to auth routes
app.post('/api/auth/login',    authLimiter, loginController);
app.post('/api/auth/register', authLimiter, registerController);
app.post('/api/auth/forgot',   authLimiter, forgotController);
💡

Pattern: Use a loose global limiter (100 req/15min) on all routes as a baseline, then apply a tight limiter (5–10 req/15min) on specific sensitive routes like login, register, and password reset.

03
Configuration Options
Every key option and what it does
OptionDefaultWhat it doesWhen to use
windowMs 60000 (1 min) The time window in milliseconds. Resets the counter after this duration. Always set explicitly. 15min for auth, 1min for APIs.
max 5 Max number of requests per IP per window. Requests above this get 429. 100 for general API, 5–10 for auth routes.
message "Too many requests" Response body when limit is hit. Can be a string, object, or function. Return JSON so clients can parse it programmatically.
statusCode 429 HTTP status code returned when limit is exceeded. Leave at 429 — it's the correct semantic status.
keyGenerator req.ip Function that returns the unique key for a client. Default is IP address. Use req.user.id for per-user limits on authenticated routes.
skip undefined Function returning true to skip rate limiting for certain requests. Whitelist your own server IPs, health checks, or internal services.
skipSuccessfulRequests false If true, only failed requests (4xx/5xx) count toward the limit. Great for login — only failed attempts count toward the brute-force limit.
store MemoryStore Where request counts are stored. Default is in-process memory. Must change to Redis in production with multiple server instances.

keyGenerator Examples

Custom key strategies 
// Default — limit by IP address
keyGenerator: (req) => req.ip,

// Per authenticated user (after auth middleware)
keyGenerator: (req) => req.user?.id || req.ip,

// Per API key (for developer APIs)
keyGenerator: (req) => req.headers['x-api-key'] || req.ip,

// Combine IP + route (different limits per endpoint)
keyGenerator: (req) => `${req.ip}:${req.path}`,

skip — Whitelist Trusted IPs

Whitelisting 
const TRUSTED = ['127.0.0.1', '10.0.0.1']; // internal IPs

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  skip: (req) => TRUSTED.includes(req.ip), // trusted IPs skip the limiter
});
04
Redis Store — Why It Matters
The default in-memory store breaks in production — here's the fix
⚠️

The default MemoryStore only works for single-process apps. If you run 4 server instances (4 CPUs, PM2 cluster, Kubernetes pods), each instance has its own counter. A bot can make 100 requests to each instance — 400 total — and never get blocked. Your rate limit is useless.

Bot sends 400 requests (100 to each of 4 instances):

Instance 1: 100 req → limit hit (but only knows about its own 100)
Instance 2: 100 req → limit hit (doesn't know about Instance 1)
Instance 3: 100 req → limit hit
Instance 4: 100 req → limit hit

Bot got 400 requests through. Rate limiting was completely bypassed. ✗

With Redis store (shared counter):
    ↓
All 4 instances share one counter in Redis → 429 after 100 total ✅
Step 1 — Install Redis store
Terminal 
npm install rate-limit-redis ioredis
Step 2 — Wire Redis as the store
app.js — production-ready limiter 
import rateLimit  from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';
import Redis      from 'ioredis';

const redisClient = new Redis({
  host: process.env.REDIS_HOST,
  port: 6379,
});

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  standardHeaders: true,
  legacyHeaders: false,
  store: new RedisStore({
    sendCommand: (...args) => redisClient.call(...args),
    prefix: 'rl:', // all keys prefixed rl:<ip> in Redis
  }),
});

app.use(limiter);

Store Comparison

StoreUse caseProduction ready?
MemoryStore (default) Single process, local dev only No — lost on restart, doesn't share across instances
RedisStore Any multi-instance production environment Yes — shared, persistent, fast ✓
PostgreSQL store Already have Postgres, don't want Redis Works, but slower than Redis for this use case
05
Response Headers
What Express sends back so clients know their current rate limit status

When standardHeaders: true is set, Express adds these headers to every response so clients can track their own rate limit status:

HeaderExample ValueWhat it tells the client
RateLimit-Limit 100 Maximum requests allowed in this window
RateLimit-Remaining 73 Requests left before being blocked in this window
RateLimit-Reset 1716400800 Unix timestamp of when the window resets (counter goes back to 0)
Retry-After 843 Seconds until the client can retry (sent only on 429 responses)
💡

Well-behaved API clients (and your own frontend) should read RateLimit-Remaining and slow down requests before hitting the limit, rather than hammering until they get a 429.

06
DDoS Defence — Layered Approach
Express alone cannot stop a real DDoS — defence must be layered from the cloud inward
⚠️

Express is the last line of defence, not the first. A real DDoS can send millions of requests per second. By the time traffic reaches your Node.js process, it's too late. Defence must start at the network edge — before traffic reaches your server at all.

1
Cloud-level protection (CDN / Shield)
Absorbs and filters malicious traffic before it reaches your server. Cloudflare free tier handles most real-world attacks. AWS Shield, GCP Cloud Armor, Akamai for enterprise.
Most powerful layer — handles volumetric floods
2
Nginx reverse proxy
Nginx sits in front of Express and rate-limits at a lower, more efficient level. Uses limit_req_zone and limit_conn_zone directives to throttle per-IP before Node.js is even involved.
Connection-level protection — cheaper than Express middleware
3
Express middleware stack
helmet for secure headers, body size limits, express-slow-down as a speed bump, express-rate-limit for hard cutoffs, hpp to prevent HTTP Parameter Pollution.
Application-level protection — where your rate limits live
4
Suspicious traffic detection
Filter known malicious user agents (sqlmap, nikto, masscan). Reject requests with no User-Agent header — a hallmark of raw bots. Block tor exit nodes if relevant.
Bot filtering — stops automated scanners early
5
Process resilience (cluster / PM2)
Run one Node.js worker per CPU core. If traffic spikes, all cores engage. PM2 or Node cluster ensures even a traffic spike doesn't idle available hardware.
Resilience — stay alive even under load

Full Express Middleware Stack (production)

app.js — complete protection stack 
import express    from 'express';
import helmet     from 'helmet';
import rateLimit  from 'express-rate-limit';
import slowDown   from 'express-slow-down';
import hpp        from 'hpp';

const app = express();

// Layer 1 — Secure HTTP headers
app.use(helmet());

// Layer 2 — Limit payload size (prevents large body attacks)
app.use(express.json({ limit: '10kb' }));

// Layer 3 — Prevent HTTP Parameter Pollution
app.use(hpp());

// Layer 4 — Speed bump (gradually slow down before hard block)
const speedLimiter = slowDown({
  windowMs: 15 * 60 * 1000,
  delayAfter: 50,   // start slowing after 50 requests
  delayMs: () => 500, // add 500ms delay per extra request
});
app.use(speedLimiter);

// Layer 5 — Hard rate limit
app.use(rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
}));

// Layer 6 — Block suspicious user agents
app.use((req, res, next) => {
  const ua = req.headers['user-agent'] || '';
  const blocked = ['sqlmap', 'nikto', 'masscan', 'nmap'];
  if (!ua || blocked.some(b => ua.toLowerCase().includes(b))) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  next();
});

Nginx Rate Limiting Config

nginx.conf 
# Define rate limit zones in http block
http {
  # 10MB shared memory zone, 10 req/sec per IP
  limit_req_zone  $binary_remote_addr zone=api:10m rate=10r/s;

  # Max 20 simultaneous connections per IP
  limit_conn_zone $binary_remote_addr zone=conn:10m;

  server {
    location /api/ {
      limit_req  zone=api burst=20 nodelay;
      limit_conn conn 20;
      proxy_pass http://localhost:3000;
    }
  }
}
07
Attack Types & Defences
Each attack type has a different fix — match the defence to the threat
Attack TypeHow it worksRight defence
Volumetric flood Millions of requests/sec from botnets, saturating bandwidth Cloudflare / CDN — absorb at the edge before reaching server ✓
HTTP flood Valid HTTP requests at high volume from distributed IPs express-rate-limit + Nginx limit_req_zone ✓
Brute-force login Thousands of password attempts on /login endpoint Tight auth limiter (5 req/15min) + skipSuccessfulRequests ✓
Slowloris Opens many connections, sends headers slowly to exhaust workers Nginx client_body_timeout + client_header_timeout ✓
Large payload attack Sends massive JSON bodies to exhaust memory/CPU express.json({ limit: '10kb' }) — reject oversized bodies ✓
Bot / scanner traffic sqlmap, nikto, masscan running automated scans User-agent filtering middleware — block known tool signatures ✓
HTTP Param Pollution Sending duplicate query params to confuse validation logic hpp middleware — keeps last value, discards duplicates ✓
💡

Quick wins with zero code: Enabling Cloudflare in front of your domain (even the free tier) eliminates the vast majority of real-world attacks before a single request reaches your server. Do this first, then add the Express middleware stack.
